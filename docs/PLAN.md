# PLAN

## Objective

現行コードを基準に、入退出管理システムを「運用可能な単一プロダクト」として完成度を上げる。ここでの完成とは、端末、API、Discord、DB、運用手順がひとつの仕様に収束し、未実装箇所とリスクが可視化されている状態を指す。

## Current Delivery Shape

- 端末アプリ、Workers API、D1 スキーマ、Discord 通知、夜間自動退出はすでに存在する
- 主要ユースケースのテストはある
- ただし管理系コマンド、運用フロー、ドキュメント整備は未完で、実装と認識のずれが起きやすい

## Milestones

### M1. Documentation Reset

- `docs/` と `AGENTS.md` を正本として整備する
- API 契約、環境変数、運用コマンド、既知の制約を文書化する
- 完了条件:
  - 実装と矛盾する主要ドキュメントがない
  - 新規参加者が docs だけで全体像を追える

### M2. Operational Hardening

- Raspberry Pi の実行前提、USB 権限、GPIO 配線、秘密情報管理を明確化する
- Discord / D1 / KV の依存先と障害時の挙動を整理する
- 完了条件:
  - 端末起動条件と API デプロイ条件が再現可能
  - 失敗時の一次切り分け手順がある

### M3. Product Completion

- `room-admin` 系の仕様を確定し、未実装状態を解消するか、正式にスコープ外へ落とす
- 新規登録許可設定の保存先と反映ルールを決める
- 完了条件:
  - コマンド定義と実装に差分がない
  - 管理権限や登録制御の動作が明確

### M4. Reliability Improvements

- 端末から API への障害耐性を見直す
- Discord 通知失敗時の影響範囲、再送方針、監視導線を決める
- 完了条件:
  - 既知の単一点障害に対する扱いが設計済み
  - 障害時のユーザー影響が説明できる

## Architecture Overview

- `crates/app`: 物理端末。Pasori 読取結果を API へ送信し、レスポンスに応じて音声と解錠を行う
- `packages/api`: アプリケーション境界。Discord Interaction、カードタッチ、夜間バッチ、D1/KV を統合する
- `crates/pasori`: 端末の低レイヤー USB/FeliCa アクセス
- Discord は通知と登録 UI を兼ねる
- D1 はユーザー、カード紐付け、入退室ログの正本
- KV は Discord ユーザー情報キャッシュに限定利用

## Delivery Order

1. 文書を最新化し、仕様の正本を固定する
2. `room-admin` をどう扱うか意思決定する
3. 運用上の前提条件と障害対応を固める
4. 必要に応じて通知失敗や端末障害への設計改善を入れる

## Risk Register

| ID  | Risk                                                     | Impact                                   | Current Handling                  | Next Action                            |
| --- | -------------------------------------------------------- | ---------------------------------------- | --------------------------------- | -------------------------------------- |
| R1  | `room-admin` がコマンド定義だけ存在し未実装              | 仕様と実装がずれる                       | 未実装レスポンスを返す            | 仕様確定か削除を決める                 |
| R2  | 非 Raspberry Pi 環境では Noop runtime で起動だけ成功する | 誤った動作確認をしやすい                 | README 上の注意のみ               | RUNBOOK と STATUS で明示し続ける       |
| R3  | API が Discord 通知送信失敗を巻き込む                    | タッチ成功が通知失敗で失敗扱いになりうる | 個別ハンドリングなし              | 通知失敗の許容方針を決める             |
| R4  | Pasori / USB / GPIO は実機依存                           | CI で完全再現できない                    | arm CI と portable runtime を併用 | 実機確認手順を固定化する               |
| R5  | 未登録 NFC コードは 4 桁かつ最大 16 回リトライ           | 衝突や登録失敗の可能性がある             | リトライで吸収                    | 必要ならコード長や生成戦略を見直す     |
| R6  | 夜間自動退出は固定 cron 前提                             | イベントや長時間利用時に誤退出しうる     | 毎日一括退室                      | 運用要件に合わせて時刻とルールを再検討 |
| R7  | 端末側は API 失敗時に再送しない                          | 一時障害で入退出記録が欠落する           | 即時エラーのみ                    | 再試行やローカルキューの要否を判断する |

## Success Criteria

- 仕様、設計、現状が相互参照可能で矛盾しない
- 実装済み機能と未実装機能が一目で分かる
- 新しい開発者が build, lint, test, deploy まで迷わず進める
